Sandbox & provider

Every task runs on a real machine of its own. This page is what that machine is, how a provider brings a fresh one up and tears it down each run, and how the framework reaches inside to do its work.

A sandbox is a whole computer

ALE's unit of environment is a sandbox: a complete Windows or Linux operating system, with the real applications installed and the task's files on disk. We give an agent full control of the machine: not only its shell and filesystem, but its graphical desktop too. That GUI access is essential, because many real-world tasks live in professional applications that cannot be driven by code and the command line alone.

Providers boot the sandbox

Each run gets a brand-new sandbox, created just for it and deleted when it ends, so no task ever inherits another's leftovers. A Provider owns that lifecycle. acquire() boots a machine from a published image and waits until it answers, then returns a SandboxHandle: the thin async surface the whole framework uses to act on the sandbox (run a command, read or write a file, upload and download). release() deletes it when the run ends, pass or fail. (A task gets its own session through open_session(), covered under Task spec.)

Two providers back the sandbox today:

ProviderBacks the sandbox withUse
gcloudA fresh GCE VM, one per run, from a published imageThe production path, many sandboxes in parallel for leaderboard runs.
staticA sandbox you already have, at a fixed addressDebugging, or wrapping a VMware / local VM you brought up by hand.

Virtual machines are heavy to provision, so ALE leans on Google Cloud as its main provider: a stable, well-supported platform that can bring up many sandboxes in parallel. Other clouds (AWS, Azure, …) and a local VMware / QEMU provider for debugging are on the roadmap, though running locally demands much more of your own machine and is hard to scale to hundreds of parallel tasks.

How the framework reaches in

The framework never touches the VM directly. Each image runs the open-source Cua computer-server, which exposes the sandbox's shell, filesystem, and GUI over a single HTTP API that is identical on Windows and Linux. The SandboxHandle is simply a client of that server, so every layer above it stays the same whatever OS a task needs. (Tasks reach the same server through cua-bench's RemoteDesktopSession.) Credit to Cua for the package.

Prebuilt images

A provider boots from an image, a disk snapshot that decides what is already on the machine. To speed experiments up and make ALE easy for the community to run, we maintain two prebuilt images on Google Cloud: ale-ubuntu22 (Linux) and ale-win10 (Windows). Each is ready to boot, carrying the OS and its professional software, the Cua computer-server, the in-sandbox agent CLIs, the Node and Python runtimes, the GUI MCP bridge, and, for many tasks, the task data itself. You copy them into your own project once, as part of setting up Google Cloud. A small image registry records each image's paths and server port, so nothing else in the framework hard-codes them.
