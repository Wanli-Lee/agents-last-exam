# ALE 错例分析 — gpt-5.5 + ale-claw(100 个失败案例)

对我们 `ale-claw + openai/gpt-5.5` 跑过的**全部失败案例**(得分 < 0.99)做根因归因,覆盖 Linux(Docker)和 Windows(GCP 虚拟机)两套沙箱。每个案例由一个独立 agent 深读(任务要求 + 实际产物 + 完整轨迹)后,按固定 schema 分类。

- **100 个失败案例** = 83 个 Linux + 17 个 Windows(去重,每个任务取最新一次运行)
- 框架:ale-claw,GUI-as-Tool 模式(computer / exec / read-write-edit / web / analyze_image / delegate_gui / memory)
- 评分:基于产物(deliverable-based),尽可能确定性判分
- 数据文件:
  - `逐case详细分析.md` — **100 例逐个详细中文归因**(任务要求/失败经过/缺失能力/证据)
  - `attributions_cn.json` — 完整中文归因(含证据、turn 引用)
  - `attributions.json` / `attributions.csv` — 原始英文版

---

## 核心结论:约 1/4 的"失败"根本不是模型的错

| 能力判定 | 数量 | 含义 |
|---|---|---|
| `yes_capability` 模型能力短板 | 38 | 真实的模型能力缺口 |
| `mixed` 能力+环境混合 | 33 | 能力缺口 + 环境等其他因素 |
| `no_environment` 非模型-环境 | 13 | 虚拟机/沙箱/部署故障(如 docker 缺失、产物下载损坏) |
| `no_eval_artifact` 非模型-评分 | 10 | 模型基本做对了,被苛刻/精确匹配的评分器判 0 |
| `no_harness` 非模型-框架 | 6 | 框架的文件传输/重布置损坏了一个正确的产物 |

**71 个是真实的模型能力问题(yes + mixed);29 个是环境/框架/评分的伪影。** 衡量真实能力差距时,这 29 个应当剔除——把它们算进去会把失败率虚高约 40%。

**非模型失败的典型例子**:`business_finance/american_option_pricing_ls` 初次运行得分 **1.0**,但框架的 CUA 产物下载在拉取 `exercise_boundary_tier2.npy` 时失败,留下了一个 `.unreadable` 占位文件;仅评测的重跑从损坏副本重布置,判了 **0.0**。模型的解是对的。

---

## 真实能力差距在哪(71 例,yes+mixed)

| 失败定位 | 数量 |
|---|---|
| **领域专业知识** | 26 |
| 长程规划 | 11 |
| 编码/实现逻辑 | 10 |
| 输出格式契约 | 8 |
| 任务理解 | 4 |
| 环境/基础设施 | 4 |
| 自检/验证 | 3 |
| GUI视觉感知 | 3 |
| 过早放弃 | 2 |

**头号真实差距是领域专业知识(26)**:模型工具用得对、流程对,但**专业判断错**——逆向二进制时提交了诱饵 config、BPMN 重设计违反角色分离规则、选错了治理压制规则的解读、计算化学方法学错误。这不是工具使用或 GUI 问题,是专业知识。

第 2/3 位的差距(100+ 轮的长程规划、编码逻辑)有一个反复出现的模式:**模型产出一个 schema 合法但空心/错误的产物,没有验证就宣布 DONE**——只填最新年份、硬编码常量、用宽松 regex 自检而不是用评分器自己的方法。

## 修复杠杆(全部 100 例)

| 修复方向 | 数量 |
|---|---|
| 更强底座模型 | 62 |
| 修环境 | 18 |
| 修框架 | 11 |
| 评分噪声/不可修 | 5 |
| 改 prompt | 4 |

**29 个案例不用动模型就能修(修环境 18 + 修框架 11)** —— 修好 Docker 部署和 CUA 产物下载/重布置流程就能直接挽回。这是最便宜的提分点。

---

## Linux vs Windows

| | 案例数 | 真实能力问题 | 环境/框架/评分 |
|---|---|---|---|
| Linux | 83 | 55 | 28 |
| Windows | 17 | 16 | 1 |

**Linux 几乎承担了所有非模型失败(29 个里 28 个)。** Linux 的 Docker 运行碰到了远多于 Windows 的部署/传输问题(docker-in-docker 缺失、产物回收下载错误、仅评测重布置损坏)。Windows 的失败几乎纯是能力缺口。→ **Linux 的框架/环境是最便宜的提分点;Windows 的失败需要更强的模型。**

---

## 交织(GUI×CLI)

只有 **14/100** 个失败发生在 GUI×CLI 交织的边界。绝大多数 ALE 失败是单通道的(纯 CLI/领域/编码)。这与 ALE 以 CLI 为主导一致——GUI 交织在这里很少是失败点(在 WeaveBench 里它要核心得多)。

---

## 该修什么,按优先级

1. **框架/环境(29 例,便宜)** — 修 Docker 部署 + CUA 产物下载/重布置损坏。多个本来正确的解纯粹因为文件传输坏掉被判 0。
2. **评分噪声(10 例 `no_eval_artifact`)** — 精确匹配/大小写脆弱的评分器把接近正确的产物判 0(如 `## modification_1:` vs 要求的 `## Modification N`)。值得审计评分器。
3. **领域知识(26 例,需更强模型/领域 SFT)** — 最硬的核心;框架修不了。
4. **自检纪律(反复出现)** — 模型在空心/未验证的产物上宣布 DONE。一个"提交前用评分器自己的方法验证"的行为(可用 RL 奖励)能挽回一批编码/格式失败。

逐例证据(turn 引用、产物文件状态、评分器输出)见 `逐case详细分析.md` 和 `attributions_cn.json`。
